# 后端双语 Mock 数据与语言锁定设计

日期：2026-07-05

## 1. 背景

前端已经支持 `zh-CN` 与 `en-US`，并在业务请求中携带
`Accept-Language`。当前 FastAPI 后端只校验并回显该请求头，默认问卷、
受访者画像、模拟回答、终止原因和分析维度仍使用中文 Mock 数据，因此英文
界面会混入中文业务内容。

现有 Mock 实现也尚未完全收口：

- `backend/app/services/mock_engine.py` 包含主要随机数据与行为；
- `backend/app/services/run_service.py` 仍包含 Mock 专属终止文案，并直接访问
  `MockEngine.rng`；
- `backend/app/services/analytics_service.py` 仍包含中文分析维度和收入标签；
- `RunService` 直接依赖具体 `MockEngine`，未来替换为真实实现时需要修改通用
  运行编排。

因此，本次不仅补齐英文数据，还要建立清晰、可替换的 Mock 边界。

## 2. 目标

- 为全部后端 Mock 业务数据提供完整的中文与英文版本。
- 只有请求头精确等于 `zh-CN` 时使用中文；`en-US`、缺失值和其他任何值均
  使用英文。
- 用户确认问卷或访谈模式后冻结前端语言，避免一次运行出现语言切换。
- 创建运行时固化语言；后续轮询、历史、分析和导出保持原始运行语言。
- 将 Mock 专属数据和行为集中到 `backend/app/mocks/`。
- 让通用运行服务依赖可替换的执行接口，而不是具体 Mock 实现。

## 3. 非目标

- 第三种界面语言。
- 自动翻译用户输入的标题、问题、选项或回答。
- 将已有运行或历史记录动态翻译成另一种语言。
- 拆分独立的问卷与访谈前端路由。
- 接入真实模型、数据库或正式任务队列。
- 本地化 HTTP 404、Pydantic 校验信息等非 Mock 协议错误。
- 改变已有 API 枚举、JSON 字段名或 CSV 协议列名。

## 4. 方案选择

采用“模式确认时冻结语言 + 运行创建时后端固化语言 + Mock 包集中管理”的
方案。

未采用以下方案：

- 仅隐藏语言按钮：界面看似锁定，但状态层仍允许修改，约束不完整。
- 以首次后端响应作为锁定点：默认模板会在模式弹窗期间异步加载，容易产生
  请求竞态，且用户无法可靠判断锁定时机。
- 拆分问卷与访谈路由：边界清晰，但会扩大前端结构重构，本期没有必要。

## 5. 前端语言生命周期

当前实际页面流程为：

```text
登录页 → 模式选择弹窗 → 问卷/访谈工作台
```

语言行为如下：

1. 登录页允许切换语言，并继续使用现有 Cookie 持久化。
2. 模式选择弹窗允许切换语言。
3. 弹窗内每次切换语言都请求对应语言的默认模板。
4. 并发模板请求只接受最后一次请求结果，较慢的旧请求不得覆盖新语言。
5. 只有当前语言的模板成功加载后，问卷与访谈模式按钮才可用。
6. 模板失败时保留模式选择弹窗并显示本地化错误，用户可重试或切换语言。
7. 用户确认模式时冻结当前语言。
8. 冻结后隐藏工作台页头的语言切换器，语言 Provider 也拒绝后续语言修改。
9. 工作台仍可保留既有模式切换能力，但不得因此解锁或更改语言。

前端冻结是用户体验约束，后端运行语言固化是数据一致性约束，两者同时存在。

## 6. 后端语言解析

新增与 HTTP 层解耦的语言类型和规范化函数：

```text
Accept-Language == "zh-CN" → zh-CN
其他所有情况                  → en-US
```

该比较为精确字符串比较，因此 `zh-cn`、`zh-CN,zh;q=0.9`、`fr-FR` 和缺失值
均解析为 `en-US`。

业务接口不再因语言头缺失或值不受支持返回 HTTP 400。响应
`Content-Language` 使用规范化后的 `zh-CN` 或 `en-US`，不回显任意原始值。
健康检查保持现有独立行为，不参与 Mock 数据选择。

## 7. 后端结构与职责

目标结构：

```text
backend/app/
├── locales.py
├── mocks/
│   ├── __init__.py
│   ├── catalog.py
│   └── engine.py
└── services/
    ├── analysis_labels.py
    ├── simulation_engine.py
    ├── template_provider.py
    ├── run_service.py
    └── analytics_service.py
```

职责如下：

- `locales.py`
  - 定义 `Locale`；
  - 实现请求语言规范化；
  - 不包含 Mock 数据。
- `mocks/catalog.py`
  - 保存结构一致的中英文 Mock 数据目录；
  - 提供默认模板、画像池、回答模板、终止文案和分析显示标签。
- `mocks/engine.py`
  - 实现所有随机画像、回答、情绪、延时和终止逻辑；
  - 只从所选语言目录读取用户可见 Mock 文案；
  - 不让随机数生成器或延时参数泄漏到通用运行服务。
- `services/simulation_engine.py`
  - 定义运行编排所需的执行接口；
  - 不引用 `app.mocks`；
  - 为后续真实实现提供替换点。
- `services/template_provider.py`
  - 定义按语言取得默认模板的接口；
  - 模板 API 只依赖该接口，不直接导入 `app.mocks`。
- `services/analysis_labels.py`
  - 定义统计服务所需的维度与分组标签接口；
  - Analytics Service 只依赖该接口，不直接导入 `app.mocks`。
- `services/run_service.py`
  - 维护通用运行状态机和异步任务编排；
  - 仅依赖执行接口；
  - 不包含 Mock 文案、随机规则或对具体 Mock 类的引用。
- `services/analytics_service.py`
  - 保留通用统计计算；
  - 通过注入的标签接口，根据运行固化的语言取得分析显示标签；
  - 不硬编码中文或英文 Mock 标签。
- `main.py`
  - 作为唯一组合根创建 Mock Engine、模板 Provider 和分析标签 Provider，并
    注入通用 API 与 Service；
  - 未来切换真实实现只需更换组合根注入。

`backend/app/services/mock_engine.py` 在迁移完成后删除。

## 8. 双语 Mock 数据范围

中英文目录必须完整覆盖：

- 默认问卷标题与描述；
- 默认问题、选项和量表题文案；
- 默认受访者性别、职业、城市和收入；
- 姓名、昵称、教育、婚姻状态和画像标签；
- 默认职业、城市与收入回退值；
- 量表、选择和文本回答的自然语言模板；
- 受访者主动退出对话文案；
- 主动终止、低质量回答和消极态度终止原因；
- 分析维度名称、收入标签前缀和分组标签。

以下值保持协议稳定，不翻译：

- ID、时间、数值和 Emoji；
- `survey`、`interview`、`completed`、`positive` 等枚举；
- JSON 字段名和 CSV 列名；
- 用户自行输入的内容。

英文默认模板和英文画像必须使用自然英文，而不是逐词机械替换。中文数据保持
现有可见内容和行为。

## 9. 运行、历史与分析数据流

`RunSnapshot` 和 `SurveyHistoryRecord` 增加 `locale` 字段。

创建运行时：

1. HTTP 依赖规范化 `Accept-Language`。
2. API 将规范化语言显式传给 `RunService.create_run`。
3. `RunService` 把语言写入初始快照。
4. 异步执行任务始终从快照读取语言。
5. Mock Engine 使用该语言生成画像、回答和终止内容。
6. Analytics Service 使用该语言生成维度和分组显示标签。

后续读取时：

- 轮询请求头只决定本次 HTTP `Content-Language`，不得改写快照语言；
- 数据正文保持运行创建时的语言；
- 即使后续请求携带另一语言，也不得翻译或重新生成运行数据；
- 保存历史时复制运行语言；
- 查询历史分析时使用历史记录语言；
- JSON 与 CSV 导出保留原始运行或历史数据语言。

用户手工修改的配置字段始终按原文保存。即使它们与运行语言不同，也不自动
翻译。

## 10. 错误与竞态处理

- 模板切换复用最新请求追踪机制，旧响应不得覆盖新响应。
- 当前语言模板未成功加载时禁止确认模式。
- 模板加载失败时不进入工作台，也不沿用另一语言的旧模板。
- 缺失或未知语言头安全回退英文，不产生语言协议错误。
- 运行创建后，任何请求都不能通过请求头修改其语言。
- Mock 数据目录的中英文结构由测试约束，避免某一语言漏字段后隐式回退。

## 11. 测试与验收

### 11.1 后端单元测试

- 精确 `zh-CN` 返回中文。
- `en-US`、缺失值、`fr-FR`、大小写变体和带权重列表均返回英文。
- 中英文 Mock 目录具备相同字段结构。
- 两种默认模板的题目数量、ID、类型和量表结构一致。
- 固定随机种子下，中文与英文分别产生对应语言的画像和回答。
- 两种语言分别产生对应的主动退出和调研方终止文案。
- 分析维度、收入标签和分组标签使用运行语言。

### 11.2 后端接口与生命周期测试

- 默认模板按规范化语言返回并设置 `Content-Language`。
- 缺失和未知语言头不再返回 400。
- 英文运行包含英文模板、画像、对话、分析和终止原因。
- 以不同语言轮询已创建运行，不改变 `RunSnapshot.locale` 或正文。
- 历史记录保留运行语言。
- 历史分析和导出保持原始数据语言。
- `RunService` 不直接导入或依赖具体 Mock Engine。

### 11.3 前端测试

- Locale Provider 在未锁定时允许切换，锁定后拒绝修改。
- 模式确认调用语言锁定。
- 模式选择弹窗保留语言切换器。
- 工作台页头不再提供语言切换器。
- 当前语言模板未加载时模式按钮禁用。
- 快速切换语言时旧模板响应不能覆盖新响应。
- 所有业务请求继续携带当前冻结语言。

### 11.4 全量验证

- 后端 `pytest`；
- 前端 Node 测试；
- TypeScript 类型检查；
- ESLint；
- Next.js 生产构建；
- 中英文 Mock 文案静态扫描；
- Git diff 与未跟踪文件检查。

## 12. 完成标准

- 中文模式保持当前中文 Mock 体验。
- 英文模式的默认模板、画像、对话、分析、历史和导出均为英文 Mock 数据。
- 模式确认后前端不能再切换语言。
- 已创建运行不会因后续请求语言变化而改变正文。
- Mock 专属数据和行为只存在于 `backend/app/mocks/`。
- 通用运行服务不依赖具体 Mock 实现。
- 所有既有和新增验证通过。
