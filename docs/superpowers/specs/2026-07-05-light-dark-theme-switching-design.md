# 明暗主题切换设计

## 背景

当前前端只提供暗色主题：根布局把 `dark` 类硬编码到 `<html>`，`app/globals.css` 的根变量与 `.dark` 变量也使用同一套暗色值。项目已经安装 `next-themes`，并已有一个薄封装的 `ThemeProvider`，但尚未接入根布局。

本次改动为项目新增明亮主题，并允许用户在界面右上角切换明亮与暗色主题。用户提供的截图只作为明亮主题的风格与颜色参考，不要求还原截图中的布局或内容。

## 目标

- 首次访问默认使用明亮主题。
- 用户可以在登录页、模式选择弹窗和工作台右上角切换主题。
- 用户选择保存在当前浏览器中，刷新和后续访问继续使用上次选择。
- 主题切换只改变前端 UI 与组件颜色。
- 切换主题时不改变表单、模式、问卷、模拟状态、分析结果或任何后端数据。
- 现有暗色主题继续可用，并保持当前视觉方向。

## 非目标

- 不调整页面布局、内容层级、文案、业务交互或数据结构。
- 不新增系统主题跟随模式。
- 不修改后端接口、认证、模拟流程或分析逻辑。
- 不追求逐像素还原参考图片。
- 不重构与主题无关的组件。

## 技术方案

采用项目已经依赖的 `next-themes`，通过根元素的 `class` 在 `light` 与 `dark` 之间切换。所有业务组件继续消费 Tailwind 的语义色类，主题差异集中维护在全局 CSS 变量中。

根布局使用现有 `ThemeProvider` 包裹应用，配置如下：

- `attribute="class"`：在 `<html>` 上维护主题类，与现有 Tailwind `dark` 变体及图表主题机制兼容。
- `defaultTheme="light"`：没有历史选择时使用明亮主题。
- `enableSystem={false}`：只提供明确的明亮和暗色选择，不随操作系统变化。
- `disableTransitionOnChange`：切换瞬间关闭全局颜色过渡，避免大面积组件出现不一致的动画闪烁。

`next-themes` 使用浏览器本地存储恢复历史选择。根 `<html>` 添加 `suppressHydrationWarning`，避免服务端无法预知本地主题时产生无意义的 hydration 警告。

## 颜色系统

### 明亮主题

`:root` 作为默认明亮主题，采用以下视觉方向：

- 页面背景使用近白的冷灰蓝。
- 卡片与弹出层使用白色或非常浅的灰蓝。
- 正文使用深蓝黑，次要文字使用中性灰蓝。
- 边框与输入框使用浅灰蓝，保证区域边界清晰但不过重。
- 主色保留品牌蓝，强调色保留青色。
- 成功、警告、错误与图表数据色继续表达原有语义。
- 阴影、半透明背景和模糊效果通过现有组件样式自然适配，不重新设计布局。

### 暗色主题

`.dark` 保留当前暗色变量及现有蓝青品牌方向。只在验证发现对比度或组件状态不清晰时，对对应语义变量做最小修正，不进行视觉重做。

### 组件约束

- 业务组件不得通过 `theme === "dark"` 分支选择业务展示内容。
- 页面级组件不得新增硬编码的主题背景或文字颜色。
- 图表系列颜色继续表示数据语义，不因主题切换改变数据含义；坐标、网格、提示框等结构色通过语义变量适配。
- 现有少量固定色只在其承担状态或数据语义时保留；若固定色承担普通表面或文字作用，则改用语义色。

## 主题切换组件

新增独立的客户端组件 `ThemeSwitcher`，职责仅包括：

- 通过 `useTheme` 读取当前主题。
- 调用 `setTheme("light")` 或 `setTheme("dark")`。
- 展示“明亮／暗色”分段按钮及太阳、月亮图标。
- 为当前项提供可见选中态与 `aria-pressed` 状态。
- 在组件挂载前使用尺寸一致的占位外观，避免服务端渲染与客户端恢复主题时发生布局抖动。

切换器使用项目现有按钮、边框、背景和文字语义色，支持键盘 Tab 聚焦与 Enter/Space 激活。无障碍名称随当前界面语言显示中文或英文，不引入新的业务状态。

## 放置位置

- 登录页：页面右上角，与语言切换器并列。
- 模式选择弹窗：弹窗右上角，与语言切换器并列。
- 工作台：顶部右侧，放入现有模式与状态控制区域附近。

小屏幕下允许右上角控制区域按现有 flex 规则换行，不改变主要工作区布局。分段按钮保持紧凑尺寸，避免挤压标题和模式选择。

## 状态与数据流

1. 首次访问时，ThemeProvider 使用 `light`。
2. 用户选择明亮或暗色后，`ThemeSwitcher` 调用 `setTheme`。
3. `next-themes` 更新 `<html>` 的主题类并写入本地存储。
4. CSS 语义变量随主题类变化，现有组件自动重绘颜色。
5. 刷新或再次访问时，`next-themes` 在应用交互前恢复保存的主题。

主题状态与问卷配置、模式选择、模拟运行、历史记录、分析数据及语言状态相互独立，因此主题切换不会触发业务请求或重置业务状态。

## 异常与降级

- 如果浏览器禁止本地存储，本次会话仍可切换主题，但后续访问可能回到默认明亮主题；不影响应用使用。
- 如果主题尚未在客户端恢复，切换器显示尺寸稳定的占位状态，不错误宣称某个主题已被选中。
- 如果某个图表或第三方组件不完全响应 CSS 变量，只修正其展示层配置，不改变传入数据。

## 测试与验收

### 自动化验证

- 测试根布局不再硬编码暗色类，并正确配置 ThemeProvider 的默认主题与 class 属性。
- 测试 ThemeSwitcher 的两个选项、选中状态和主题切换调用。
- 测试主题切换器出现在登录页、模式选择弹窗和工作台。
- 运行现有前端测试、类型检查和生产构建，确认业务逻辑与构建行为未受影响。

### 浏览器验证

- 清除主题存储后打开登录页，确认默认明亮主题。
- 在登录页切换暗色并刷新，确认选择被保留。
- 登录后确认模式选择弹窗仍为已保存主题，并可再次切换。
- 选择模式进入工作台，确认切换主题不改变当前模式、表单字段、问卷数据或分析展示。
- 在明暗主题下分别检查登录页、模式弹窗与工作台。
- 检查移动、平板和桌面布局，确认切换器不遮挡、不溢出且可键盘操作。
- 检查输入框、按钮、下拉菜单、弹窗、状态色、空状态和图表的可读性。
- 检查页面加载与主题切换时没有明显主题闪烁或布局跳动。

## 完成标准

当首次访问默认明亮、三处入口均可切换、选择可持久化、两套主题下主要界面可读，并且主题切换不会改变任何业务状态或数据时，本功能完成。
