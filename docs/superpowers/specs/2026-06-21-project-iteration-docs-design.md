# 项目迭代文档体系设计

## 目标

为 Survey Agent Simulator 建立可持续维护的项目说明与迭代台账，使团队能准确了解系统现状、能力边界、当前问题、近期优先级及每轮迭代的验证结果。

## 文档职责

### `README.md`

README 面向首次接触项目的开发者与协作者，保留项目定位、现有能力、架构、运行方式，并增加：

- 当前阶段与明确的能力边界；
- 当前最迫切的迭代模块及排序依据；
- 项目迭代文档入口；
- 每次迭代必须同步更新文档的维护约定。

README 只保存路线图摘要，不复制完整问题池和迭代记录。

### `docs/PROJECT_ITERATION.md`

该文件是项目迭代状态的唯一完整事实源，包含：

- 文档维护规则与更新时机；
- 基于代码和浏览器实测的当前现状；
- 已实现能力与尚未实现的边界；
- 按影响和依赖关系排序的问题池；
- P0、P1、P2 路线图及阶段验收标准；
- 当前最迫切模块的范围、任务拆分和完成定义；
- 每轮迭代记录模板与持续追加的变更记录。

体验评价与布局评价文件作为评估依据保留，不承担持续台账职责。

## 优先级设计

当前迭代顺序为：

1. 响应式布局与滚动模型：解决移动端 390 px 下 491 px 横向溢出、桌面端文档高度超出视口、窄桌面工作区拥挤等基础可用性问题。
2. 研究轮次、配置快照与保存/导出可信度：建立可复现的数据组织方式和明确的操作反馈。
3. 样本方案与分析口径：支持自定义属性、配额预览，并明确总体、筛选、分组结果的有效样本数。
4. 研究设计与证据链：增加设计检查、试跑、异常管理、结果下钻和研究假设沉淀。

响应式布局排在第一位，因为它是当前代码和实测共同确认、范围清晰且影响所有后续工作区改造的阻塞项；研究记录体系是产品从演示工具转向研究工作台的首要业务能力，紧随其后。

## 每轮迭代的更新流程

每次功能、修复或架构迭代必须在同一变更中：

1. 更新 `docs/PROJECT_ITERATION.md` 的现状、问题状态、路线图和迭代记录；
2. 若项目定位、核心能力、架构、运行命令或最高优先级发生变化，同步更新 `README.md`；
3. 记录完成内容、验证方式、遗留问题和下一步；
4. 不把规划中的能力描述成已实现能力；
5. 以实际代码、构建结果和浏览器验证作为状态更新依据。

## 验证

- 检查 README 中存在迭代文档链接、当前状态摘要和维护约定；
- 检查迭代台账包含现状、问题、路线图、当前焦点、验收标准和记录模板；
- 检查两份文档的优先级和能力边界一致；
- 检查所有本地链接有效、Markdown 标题层级与表格格式正确；
- 通过 `npm run build` 确认文档变更未伴随项目构建回归。
