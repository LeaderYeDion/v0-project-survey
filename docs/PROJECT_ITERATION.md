## 文档定位与维护规则

本文是项目迭代状态的唯一完整事实源，用于持续记录系统现状、能力边界、问题优先级、阶段路线图、验收标准和每轮迭代结果。

每次功能、修复或架构迭代都必须在同一变更中：

1. 更新本文的现状、能力边界、问题状态和迭代记录；
2. 记录本轮目标、实际完成内容、验证方式、遗留问题和下一步；
3. 若项目定位、核心能力、架构、运行命令或最高优先级发生变化，同步更新根目录 `README.md`；
4. 以实际代码、构建结果和浏览器验证为依据，不把规划中的能力写成已实现能力；
5. 保留历史评估文档作为决策依据，不在其中继续维护实时项目状态。

状态标记：`未开始`、`进行中`、`已完成`、`已阻塞`、`暂缓`。每完成一项，应更新对应状态和完成日期，不删除仍具有追溯价值的问题。

## 当前项目现状

### 产品状态

项目当前是一套 AI 调研流程的可交互前端原型，目标用户可配置问卷与受访者画像，运行 mock 访谈或问卷模拟，并查看个体回答、聚合统计和人口维度分析。它适合演示产品闭环、验证信息架构和开展探索性体验评估，尚不具备严肃研究所需的真实 AI 服务、持久化、版本管理和可复现运行基础设施。

### 已实现能力

| 模块 | 当前实现 |
| --- | --- |
| 调研模式 | 支持问卷与访谈双模式；问卷强调批量结果，访谈强调受访者列表与对话详情 |
| 调研配置 | 支持标题、描述、响应时间、文本/选择/量表题，以及表单与 JSON 两种编辑方式 |
| 受访者配置 | 支持性别、年龄段、职业、城市、收入与人数的固定字段配置 |
| 模拟执行 | 前端按受访者、按问题顺序调用 mock 服务，实时更新进度、会话、情绪和分析数据 |
| 结果查看 | 支持问卷单份结果、访谈完整对话、问题分析、人口画像和完成/终止状态 |
| 分析能力 | 支持性别、年龄、职业、城市、收入等维度筛选，以及多维 group by |
| 保存与回放 | 可把当前结果保存到模块内存，并在当前页面生命周期内打开历史记录 |
| 导出 | 可在浏览器中下载 JSON 或 CSV 文件 |
| 技术栈 | Next.js 16.1.6、React 19.2.4、TypeScript、Tailwind CSS、ShadCN/Radix UI、Recharts |

### 实测基线

- 2026-06-21 桌面端 1280 × 720：文档宽度为 1280 px，无横向溢出；文档高度约 1259 px，主区域固定视口高度与子内容整页滚动并存。
- 2026-06-21 移动端 390 × 844：文档宽度为 881 px，横向溢出约 491 px；主区域仍为横向 `flex`，结果和分析区域默认落在屏幕外。
- 当前生产构建 `npm run build` 通过。项目存在自定义 Babel 配置警告，构建过程跳过类型校验，后续需单独治理。

## 当前能力边界

以下能力尚未实现，不能在产品介绍或演示中作为现有功能承诺：

- `lib/survey-api.ts` 目前只是 mock 服务的统一代理层，没有真实 AI 模型或后端 API；
- 历史记录保存在 `mockHistoryStorage` 模块内存中，刷新页面或重启进程后丢失；
- 没有“项目—研究轮次—配置版本”的数据模型，也没有不可变运行快照；
- 没有登录、权限、协作、服务端任务队列或后台运行通知；
- 运行过程没有完整的暂停、继续、取消、仅重试失败样本和预计剩余时间能力；
- 受访者字段固定，尚不支持自定义政策属性、字段类型、基础配额或交叉配额；
- 没有运行前的问卷/访谈设计检查、样本覆盖检查或试跑到正式轮次的流程；
- 分析界面尚未严格区分总体、当前筛选和分组比较口径，也缺少统一的有效样本数与小样本警告；
- 保存和导出缺少稳定的命名、范围选择、成功/失败反馈及可追溯记录；
- 当前多栏布局没有响应式降级，移动端无法在正常单栏视口内完成完整流程；
- AI 模拟结果只适合探索、预实验和假设生成，不能替代具有代表性的真实调查。

## 问题与优先级

| 优先级 | 模块 | 状态 | 当前问题 | 目标与完成标准 |
| --- | --- | --- | --- | --- |
| P0-A | 响应式布局与滚动模型 | 未开始 | 移动端横向溢出；桌面端整页与栏内滚动混用；窄桌面访谈区拥挤 | 五个目标宽度无非预期横向滚动；移动端可完成完整核心流程；桌面端关键状态和主操作始终易找 |
| P0-B | 研究轮次、配置快照与保存/导出 | 未开始 | 历史仅存内存，刷新丢失；没有项目、轮次和版本；操作反馈薄弱 | 每次运行绑定不可变配置快照；记录可再次打开；保存/导出有明确范围、文件名与结果反馈 |
| P0-C | 分析口径与异常解释 | 未开始 | 总体、筛选、分组结果层级不清；终止只有数量；问卷仍出现访谈语义 | 所有分析明确显示口径和有效样本数；终止原因可汇总和下钻；模式文案一致 |
| P1-A | 自定义样本方案 | 未开始 | 受访者字段固定，无法表达政策相关属性与配额 | 支持自定义字段、枚举值、基础/交叉配额、结构预览和覆盖缺口提示 |
| P1-B | 研究设计与运行控制 | 未开始 | 没有设计检查、试跑、暂停、取消和失败重试 | 可在正式运行前发现题目与样本问题；运行过程可控，异常可处理 |
| P1-C | 发现到证据的工作流 | 未开始 | 聚合结果难以下钻到个体证据，不能标注和沉淀假设 | 支持结果下钻、搜索、批注、收藏、异常清单和待验证假设 |
| P2 | 对比、协作、模板与性能 | 暂缓 | 缺少多轮对比、团队协作、模板库和大样本优化 | P0/P1 稳定后再建设轮次对比、报告协作、模板复用、后台任务与渐进加载 |

在 P0 和 P1 完成前，暂缓增加装饰性图表、实时动画、更多拟人化头像/标签，以及无法追溯到证据的自动结论。

## 当前最迫切的迭代模块

### P0-A：响应式布局与滚动模型

这是下一轮实施目标。它的范围清晰、已有实测基线，并会影响后续“设计—运行—分析—记录”工作区改造，因此优先于继续叠加业务面板。

#### 目标布局

- `<768 px`：单栏工作区；配置、运行/结果、分析通过顶部 Tabs 或底部导航切换；顶部信息分行，弹窗宽度受视口约束。
- `768–1199 px`：保留中心工作区，将配置和分析放入可开合 Sheet/Drawer；访谈受访者列表允许收起。
- `>=1200 px`：保留桌面三栏工作台；访谈模式允许列表窄栏，但保证对话区最小可读宽度。

#### 实施范围

1. 为页面骨架增加明确断点与 `min-w-0`、`min-h-0`、`overflow-hidden` 约束；
2. 统一滚动模型：桌面骨架固定于视口，各任务栏独立滚动；
3. 将左栏启动按钮做成栏内 sticky footer，保留配置上下文；
4. 为移动端提供配置、结果、分析的可达导航，不隐藏核心任务；
5. 处理顶部栏、模式选择弹窗、长标题、长问题和长对话的换行与截断；
6. 保持问卷和访谈两种模式的数据与交互逻辑不变，仅调整布局和导航承载方式。

#### 完成定义

- 390、768、1024、1280、1440 px 宽度下无非预期横向滚动；
- 390 px 下无需横向拖动即可完成配置问卷、启动模拟、查看结果和查看分析；
- 桌面端任一栏滚动时，模式、运行状态和主操作仍易于找到；
- 真实长标题、长问题和长对话不会遮挡按钮或撑破容器；
- 问卷和访谈模式在目标断点下都通过浏览器交互验证；
- `npm run build` 通过，控制台无新增 error。

#### 下一模块

P0-A 完成后进入 P0-B：先设计项目、研究轮次和配置快照的数据结构，再替换内存历史存储，并补齐保存与导出的反馈闭环。

## 分阶段路线图

### P0：可信、可控、可复现

目标：用户能够在常见设备上完成一轮研究，清楚确认运行配置、结果口径与异常，并再次打开同一结果。

1. 完成响应式布局与统一滚动模型；
2. 建立项目、研究轮次、配置版本和不可变运行快照；
3. 接入持久化，补齐自动保存、历史、导出和失败反馈；
4. 统一问卷/访谈的模式语义、文案和状态隔离；
5. 增加终止/失败原因、暂停、取消和重试；
6. 规范总体、筛选和分组的口径展示及有效样本数。

阶段验收：用户可以创建、受控运行、保存、重新打开并解释一轮调研；刷新页面不会丢失已保存记录；所有主要结果能说明当前样本口径。

### P1：研究设计与证据发现

目标：系统不仅运行模拟，还能帮助研究者发现设计问题、样本缺口和待验证假设。

1. 支持自定义受访者属性、基础/交叉配额与覆盖预览；
2. 支持政策研究模板、问卷/提纲导入和运行前设计检查；
3. 增加小样本试跑，以及从试跑复制为正式轮次；
4. 增加小样本、空分组、覆盖缺口和终止偏差提示；
5. 支持标准化组间比较与从图表下钻到个体证据；
6. 支持个体回答搜索、标注、收藏和假设清单；
7. 生成研究摘要、异常清单与下一轮建议。

阶段验收：用户能在系统内回答“发现了什么、证据在哪里、有什么风险、下一轮真实调研要验证什么”。

### P2：协作与规模化

目标：支持研究组持续使用和多项目管理。

1. 多轮运行、问卷版本和样本方案对比；
2. 研究者批注、分享、权限和报告协作；
3. 模板库、变量字典和配额库；
4. 后台任务、批量项目和完成通知；
5. 完整数据字典与更多报告格式；
6. 大样本性能、增量分析与渐进式加载。

## 统一验收原则

- **事实一致**：文档中的已实现能力必须能从代码或运行结果验证；
- **任务闭环**：验收以用户能否完成目标任务为准，不以新增组件数量为准；
- **口径明确**：数据结果必须说明范围、状态和有效样本数；
- **异常可解释**：失败、终止、空数据和小样本必须有可见反馈；
- **可复现**：影响结果的配置、版本、时间和运行设置应可追溯；
- **多视口验证**：涉及界面的迭代至少验证 390、768、1024、1280、1440 px；
- **工程验证**：每轮至少运行相关自动化检查和 `npm run build`，并记录已知警告；
- **文档同步**：每轮提交前完成本文更新，必要时同步 README。
